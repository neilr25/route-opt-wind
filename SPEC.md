# Technical Specification: SGS Route Optimization Engine

## 1. Project Goal
To build a self-hosted routing service that replaces static distance tables (AtoBviaC) with a dynamic, wind-aware optimization engine. The system will calculate a "Standard" route and an "Optimized" route, comparing the two to quantify fuel/time savings based on ship-specific performance polars.

## 2. System Architecture
The system follows a **Linear Pipeline**:
`Input (Ports/Speed)` $\rightarrow$ `Baseline (ATOBVIAC)` $\rightarrow$ `Mesh Generation` $\rightarrow$ `Costing (Weather + Polars)` $\rightarrow$ `Optimization (A*)` $\rightarrow$ `Visualization`.

### A. Data Layer (The "Lake")
*   **Weather Data:** ERA5/Open-Meteo daily wind speed and direction stored in **Apache Parquet** files, queried via **DuckDB**.
*   **Performance Data (Polars):** Ship-specific performance curves (provided via Excel/Supabase).
    *   *Input:* Wind Speed, Wind Direction.
    *   *Output:* Power/Speed efficiency (Power with wing vs. Power without wing).
*   **Land Mask:** Use the `global-land-mask` Python library or Natural Earth shapefiles to prevent routes from crossing land.

### B. The Routing Logic
1.  **Baseline Generation:** Use the `ATOBVIAC` library to generate the "Standard" shipping lane path between the start and end ports.
2.  **Dynamic Mesh Construction:**
    *   Create a "Corridor" (e.g., $\pm 200\text{nm}$) around the baseline route.
    *   Divide the corridor into a grid of nodes.
    *   **Temporal Slicing:** The mesh is processed in daily increments. The distance reachable in 24 hours (based on ship speed) defines the "active" slice of the mesh for that day.
3.  **Cost Function (The "Secret Sauce"):**
    *   For every node in the mesh, calculate the **Wind Angle** relative to the ship's heading.
    *   Cross-reference this angle and the wind speed against the **SGS Polars**.
    *   **Cost = (Distance to next node) $\times$ (Power Required from Polars)**.
    *   If node = Land, Cost = $\infty$.
4.  **Optimization:** Use the **A* (A-Star) Algorithm** (via `NetworkX`) to find the path from Start to End that minimizes the total cumulative cost.

---

## 3. Functional Requirements (The "Output")

The system must produce a final report/visualization containing:

### A. The Route Map
A geospatial plot (using `Plotly` or `Folium`) showing:
1.  **The Great Circle Route:** A direct line between ports (for baseline comparison).
2.  **The Standard Route:** The `ATOBVIAC` baseline.
3.  **The Optimized Route:** The path generated by the A* algorithm.

### B. The Savings Analysis
A comparison table calculating:
*   **Total Distance:** (Great Circle vs. Standard vs. Optimized).
*   **Estimated Power/Fuel Consumption:** Based on the Polars lookup for the entire voyage.
*   **Average Saving:** The percentage reduction in power/fuel when using the Optimized Route vs. the Great Circle/Standard route.

## 4. Implementation Stack (Recommended)
*   **Language:** Python 3.10+
*   **API Framework:** FastAPI (to mimic Open-Meteo JSON responses).
*   **Data Processing:** `Pandas`, `Polars`, `Xarray`.
*   **Database/Query:** `DuckDB` (querying Parquet files).
*   **Geospatial:** `GeoPandas`, `ATOBVIAC`, `shapely`, `global-land-mask`.
*   **Optimization:** `NetworkX` (A* implementation).
*   **Visualization:** `Plotly` (for interactive maps).

2.  **Phase 2: Baseline & Mesh.** Implement the `ATOBVIAC` call and the corridor grid generation logic.

4.  **Phase 4: The Optimizer.** Implement the A* search across the time-stepped mesh.
5.  **Phase 5: Visualization.** Create the Plotly map and the savings calculation table.

### Note on the Provided Polars Sample:
The engine must specifically look at the `power_without_wing` vs `power_with_wing` columns. The "Saving" is calculated as the difference in power required to maintain the same speed given the wind conditions at each node of the optimized route.
